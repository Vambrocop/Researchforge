# ResearchForge — 骨架阶段设计文档（Spec v0.1）

- 日期：2026-06-12
- 状态：待用户评审
- 范围：本文档只定义**骨架/MVP 阶段**。完整多领域平台是愿景，分期实现（"先骨架，后慢慢填满"）。

## 1. 背景与愿景

**一句话定位**：给 ResearchForge 一份数据，它自动读懂数据、推荐能做的研究分析、由你选定、再自动执行并出图出报告。

**核心体验循环**：
> 自动识别数据 → 自动推荐分析可能性 → 你来选 → 自动执行 + 自动作图出报告

**普适性来源**：分析的路由由**数据类型/结构**驱动，而非写死某个学科——所以天然能跨学科泛化、可"一点点加领域"。

**创新性来源**：① 跨学科方法迁移（数据驱动路由天然能把一个领域的方法用到另一个领域的数据上）；② 推荐阶段的"新颖度透镜"——挂一次文献扫描，告诉你哪些角度文献做过、白地在哪；③ 可选的"前沿监视器"跑在自动循环上，持续追踪新论文；④ "GitHub 采集器"——自动发现 GitHub 上的新 skill / 论文复现代码 / 新方法包并登记。详见 §7。

**起步领域**：农业经济学 / 通用统计学（已有强执行器）。后续扩：土壤学、生态学、微生物学、**空间/GIS 与全球宏观研究**。

**跨学科愿景**：面向"**农 · 生 · 环 · 经**"（农学 · 生物/生命科学 · 环境/生态 · 经济）的交叉组合研究——"跨学科方法迁移"（§7.1）正是为此而设。

**最终形态**：网页应用（换任何电脑/设备开浏览器即用）。
**当前形态（本期）**：可移植的 **Git 仓库 + Python 引擎**，经 Claude Code 驱动。网页是后续阶段的增量。

## 2. 架构：三层 + 三个横切机制

```
                ┌─────────────────────────────────────────┐
   数据文件 ──▶ │ ① Profiler 数据画像                       │
  (CSV/Excel)   │   输出: DataFingerprint (结构化指纹 JSON)  │
                └───────────────────┬─────────────────────┘
                                    ▼
                ┌─────────────────────────────────────────┐
                │ ② Recommender 推荐（混合路线 C）          │
                │   规则目录匹配 + LLM 排序/解释             │
                │   输出: 排序的 AnalysisRecommendation[]    │
                └───────────────────┬─────────────────────┘
                         你来选 ◀────┤
                                    ▼ (选中项)
                ┌─────────────────────────────────────────┐
                │ ③ Executor 执行                           │
                │   调对应 skill/脚本 → 表 + 图 + 报告        │
                └─────────────────────────────────────────┘

  横切 A: Skill Ingestion（随时添加）  横切 B: Portability（便携性）  横切 C: Frontier & Novelty（前沿与创新性）
```

### 各层职责与接口契约

- **① Profiler（数据画像）**：输入数据文件，输出 `DataFingerprint` —— 一份结构化指纹：变量列表与类型（连续/分类/计数/二值/日期/ID/**地理坐标**）、缺失情况、是否存在 `单位×时间` 面板结构、**是否为时间序列（单序列/面板时序）**、**是否含空间/GIS 维度（坐标/区域/栅格）**、是否存在处理组/对照组、样本量、分布概况，并**标注数据质量问题**（缺失、异常值、重复、类型/单位不一致）。**纯只读、确定性、不调 LLM。**
- **①.5 Cleaning（数据清洗，默认开启）**：基于 Profiler 标出的质量问题，按数据类型生成一份**清洗方案**供你确认；确认后执行并保留**清洗日志**，保证可复现。
- **② Recommender（混合 C）**：输入 `DataFingerprint` + **可选研究目标/问题**（解释/因果 vs 预测/预报 vs 分类/聚类——同一份数据按目标不同路由到不同方法家族，如面板数据做 DID 还是做 XGBoost 预测），输出排序的 `AnalysisRecommendation[]`，每条含：方法名、前提是否满足、预期产出、对应执行器引用、推荐理由。规则目录给候选与硬前提，LLM 负责排序、解释、补目录未覆盖的情况。**每条推荐必须透明可解释。**
- **③ Executor（可插拔多后端）**：输入选中的推荐 + 数据，经统一接口调对应后端执行器（Python / R / Stata / 深度学习框架），产出表格 + 图 + 报告。本期复用 `empirical-analysis-python`（Python）；其它后端见 §8 预留。

### 横切机制

- **A. Ingestion（随时添加，多渠道）**：`skills_inbox/` 投递 → 提炼（领域/方法/前提/产出）→ 登记进分析目录（catalog）→ 原件归档 `_processed/`。让"随时添加"成为一等公民。**三个投递渠道**：① **手动** skill（`SKILL.md`）放入 inbox；② **论文吸收**——放入 PDF/DOI，抽取其方法/数据前提/分析并登记（可顺带定位其复现代码）；③ **GitHub 采集器**（§7.4）自动发现 skill/代码/方法后入列。
- **B. Portability（便携性）**：一切进 Git 仓库；引擎依赖的 skill 通过 `setup` 脚本安装到 `~/.claude/skills/`（或仓库内打包）；换电脑 = `git clone` + `setup`。
- **C. Frontier & Novelty（前沿与创新性）**：在推荐层挂"新颖度透镜"（文献扫描标注白地），并可选地用自动循环跑"前沿监视器"持续追新论文。详见 §7。

### 实现风格：AI-agent 驱动 + 双模获取（自动 / 手动）

系统的"软推理"部分——推荐排序与解释、新颖度透镜、前沿监视、GitHub 采集、论文吸收——都可由**专门的 AI agent** 实现（复用 deep-research 多-agent 流水线、Claude Code subagents、agent 团队），而非写死规则。规则只负责硬前提与确定性步骤（画像、清洗、执行）。

**双模获取**：每个知识获取渠道（skill / 论文 / 前沿 / GitHub）都同时支持 **① AI-agent 自动**（辅助搜索与更新）与 **② 手动添加**，两者并存、随时切换。

> **算力/规模注**：用户为 **Max 计划**，可承载 agent 密集与周期搜索等较重功能；架构按"尽可能大"设计（见 §8 预留）。但**建造顺序仍是先打通一条纵切、再扩面**——目的是先验证闭环、降风险，与算力无关。

## 3. 推荐引擎：混合路线 C 怎么落地

- **规则目录（catalog）**：一组结构化条目（YAML/JSON），每条 = `{method, domain, preconditions, produces, executor_ref}`。
- **匹配**：用 `DataFingerprint` 比对每条的 `preconditions` → 得到候选集（硬前提保证不推荐数据不支持的方法）。
- **LLM / agent 层**：对候选排序、生成人话解释、并对目录未覆盖的情况给补充建议（标注"非目录项"）。可由专门 agent 实现（见 §2 实现风格）。
- **可靠性兜底**：硬前提由规则把关，LLM 只在已通过前提的候选上做软判断，降低乱推风险。

## 4. 本期范围（骨架 MVP）

**做（In Scope）**：
1. 仓库结构 + `setup` 脚本骨架（依赖与 skill 安装）。
2. `DataFingerprint` schema + Profiler（CSV/Excel 表格类）：**数据类型识别**（含时间序列检测）+ **数据质量诊断** + **基础清洗**（按诊断生成清洗方案 → 你确认 → 执行 + 日志）。
3. Catalog 数据结构 + 农业计量的**初始条目**（面板/截面/DID/固定效应/IV 等少量）。
4. Recommender（规则匹配 + LLM 排序），输出可解释推荐菜单。
5. **一条完整纵切跑通**：省份×年份面板 CSV → 画像 → 推荐 → 选「双向固定效应/DID」→ 调 `empirical-analysis-python` 执行 → 出回归表 + 图 + 简报。
6. Skill ingestion 跑通一次（投一个 skill → 进 catalog → 推荐中可见）。
7. **新颖度透镜（轻量版）**：推荐阶段对选中分析挂一次文献扫描（复用 deep-research / paper-search-mcp / nature-academic-search），输出"文献做过没、白地在哪"的提示。

**不做（Out of Scope，后续阶段）**：
- **前沿监视器**（跑在自动循环上的持续追新论文，留待后续）；
- **GitHub 采集器**（自动发现 + 读取 GitHub skill/代码/方法，留待后续；架构预留入列接口。安全红线：采集来的代码不自动执行）；
- 网页前端与云部署；
- 生态/土壤/微生物的原始数据执行器（需封装 vegan/lme4/agricolae/phyloseq 等）；
- **机器学习 / 深度学习 / 时间序列执行器**（RF·XGBoost、CNN·LSTM·Transformer、ARIMA·Prophet 等——架构预留，见 §8）；
- **R / Stata 执行器后端**（农学田间试验、生态、微生物多靠 R——架构预留，见 §8）；
- **空间 / GIS 与全球宏观分析**（空间计量、地统计、遥感栅格——架构预留，见 §8）；
- 文本/文献语料的画像；
- 大而全的多领域目录。

## 5. 技术选型

- **引擎核心**：Python 包（`profiler` / `catalog` registry / `recommender`）。
- **执行**：可插拔多后端、统一接口。本期 Python（`empirical-analysis-python`）；预留 R / Stata / 深度学习框架（见 §8）。
- **编排入口**：Claude Code 的 master skill / CLI 命令，把 ①②③ 串起来。
- **LLM 推理**：经 Claude（推荐层的排序与解释）。

## 6. 成功标准（可验证）

1. 丢一份省份×年份面板 CSV 进去 → 引擎正确识别为**面板数据**（识别出单位、时间、可能的处理变量）。
2. 列出 **≥3 个可行分析**，每个都说明"前提是否满足 + 预期产出"。
3. 选定一个 → 自动产出**回归表 + 图 + 简报**（复用 empirical-analysis-python）。
4. 在**另一台电脑** `git clone` + `setup` 后，能复现上述结果。
5. 往 `skills_inbox/` 投一个新 skill → 能被登记进 catalog 并在后续推荐中可见。
6. 选中的分析至少得到**一条"文献白地"提示**（新颖度透镜跑通一次）。
7. 上传数据后能给出**数据质量诊断 + 一份可确认的清洗方案**并执行（含清洗日志）。

## 7. 创新性：前沿感知与新颖度评估

ResearchForge 的创新性靠三个机制，分别接入不同位置：

1. **跨学科方法迁移（结构性创新）**：数据类型驱动的路由天生能把一个领域的成熟方法推荐到另一个领域的数据上（如把因果机器学习/DID 用到农学、生态数据）。这是架构本身带来的创意来源，无需额外搭建。
2. **新颖度透镜（接入 ② 推荐层）**：给出推荐分析菜单时，对候选角度做一次文献扫描，标注"文献里做过没、做到什么程度、白地在哪"。于是推荐从"哪些可行"升级为"哪些可行且新颖/可发表"。复用已有 skill：deep-research、paper-search-mcp、nature-academic-search、literature-review、research-ideation、k-dense-hypothesis。**本期做轻量版。**
3. **前沿监视器（独立、可选）**：每隔一段时间自动搜索用户领域的新论文/方法（arXiv、期刊、GitHub），过滤相关性 → 去重 → 排序后，新方法自动进 catalog、相关新发现推送提醒。**可行性已具备**——搜索栈用 deep-research / paper-search-mcp / nature-academic-search / WebSearch；定时器两种：`/loop`（会话内周期检查）或 **cron 云端定时 agent（电脑关机也照跑，适合"每天/每周自动巡"）**；`daily-paper-generator` 已是同类现成范例。它不止罗列新论文，还做**创新性研判**——结合文献计量趋势（什么在兴起、热点迁移、白地在哪）判断"什么是最新 / 值得做的方向"，由新颖度 agent 输出。**本期不做，留待后续。**
4. **GitHub 采集器（Skill / 代码 / 方法发现）**：自动在 GitHub 搜索目标领域的新 skill、论文复现代码、新方法包（按 topic / star / 更新时间排序），读取并登记进 catalog——是横切 A「随时添加」的自动化版，也是高性价比的前沿来源（公开仓库可直接 fetch / clone）。**安全边界（红线）：只自动「读取 + 登记」；执行采集来的代码必须经人工审核 / 沙箱，绝不自动运行陌生代码。本期不做，留待后续；横切 A 预留入列接口。**

**可靠性边界**：LLM + 检索的新颖度判断会漏检/误判，定位为"给研究者的线索与初筛"，由人终审，不作为发表性结论。

## 8. 架构预留（Reserved Extension Points）

为支持"先骨架、后慢慢填满"，以下都在架构里**预留接口、本期不实现**。每加一项 = 往 catalog / registry 登记，**不改三层结构**。

**执行器后端（统一接口）**
- Python ✅（本期，empirical-analysis-python）
- R（预留）— 农学田间试验（lme4 / nlme / agricolae）、生态群落（vegan）、微生物组（phyloseq / DESeq2）
- Stata（预留）— 复用 stata skill
- 深度学习框架（预留）— PyTorch / TensorFlow

**分析家族（挂进 catalog，由"研究目标"路由）**
- 计量经济学 / 统计模型 / 因果推断 ✅（本期起步，**定量**）
- 预测型机器学习（预留）— 随机森林、XGBoost / LightGBM（表格预测/分类）
- 深度学习（预留）— CNN、LSTM、Transformer（图像/序列/时序）
- 时间序列 / 预报（预留）— ARIMA、ETS、Prophet、darts / sktime
- 空间 / GIS（预留）— 空间计量、地统计（kriging）、GWR、遥感栅格、空间自相关；面向**全球宏观研究**（国别空间面板、土地利用/气候栅格）
- 生态群落（预留）— 排序、PERMANOVA、α/β 多样性
- 微生物组（预留）— DADA2 / QIIME2、差异丰度
- **定性 / 质性分析（预留）**— 主题分析、内容分析、扎根理论、质性编码（访谈/文本数据）；及**混合方法**（定量 + 定性）
- **系统性综述 / 证据综述** ✅ 已有 skill（evidence-synthesis-forge、meta-analysis-forge、environment-life-review-forge、deep-research、umbrella-review-skeptic）——可登记入 catalog，非本期纵切
- **文献计量学 / bibliometrics（预留）**— 引文网络、共被引、关键词共现、研究趋势图谱（bibliometrix / VOSviewer / CiteSpace / pybliometrics）；为新颖度透镜与前沿监视提供"趋势 / 前沿"信号

**数据模态（挂进 Profiler）**
- 表格 CSV/Excel ✅（本期）
- 空间 / GIS（预留）— 坐标 / shapefile / 栅格 / 遥感
- 文本 / 文献语料（预留）

**知识来源 / 投递渠道（横切 A）**
- 手动 skill 投递 ✅（本期）
- 论文吸收（预留）— PDF/DOI → 抽方法/前提/分析入 catalog
- GitHub 采集器（预留）— 见 §7.4
- 前沿监视器周期搜索（预留）— 见 §7.3

> **关键设计**：路由不只看**数据结构**，还看**研究目标**（解释/因果 vs 预测/预报 vs 分类）。同一份面板数据，目标"识别政策效应"→ DID/固定效应；目标"预测产量"→ XGBoost/LSTM。预留这些家族后，Recommender 才能按目标给出跨范式的菜单。
>
> **架构稳健性自检**：截至目前用户提出的 R 包、ML/DL、时序、GIS/全球宏观、论文吸收、周期搜索、定性/质性分析、文献计量、系统性综述、农生环经交叉，全部能作为"登记项"接入而**无需改动三层结构**——这是该架构成立的证据。

## 9. 风险与未决

- **R 环境便携性**比 Python 难（生态/土壤/农学执行器用 R 时要处理）——本期不涉及，后期再定。
- **深度学习 / GIS 依赖较重**（PyTorch、GDAL / 地理栈）——预留为独立可选后端，不进 MVP 基础安装，避免拖累便携性。
- **skill 打包**的体积/许可：本期用 setup 脚本安装而非内嵌，规避。
- **LLM 推荐 / 新颖度可靠性**：靠规则硬前提兜底；推荐与新颖度提示均展示依据，人来终审。
- **周期搜索成本**：云端定时 agent 跑搜索有 token/算力成本，需可配置频率与领域范围。
